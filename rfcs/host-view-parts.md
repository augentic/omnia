# Design: Host View Parts (retiring `omnia_wasi_view!`)

> Status: Design proposal — replaces per-host-crate `omnia_wasi_view!` macros with
> ordinary trait code and a trivial `Parts` projection generated by
> `#[derive(StoreContext)]`. No behaviour change; improves testability and removes
> the hardcoded `self.core.*` contract.
>
> Depends: the landed [Runtime Interface Extraction](runtime-interface.md)
> (`Core`, `#[derive(StoreContext)]`, `omnia::serve`). Owns: the `Parts` trait
> pattern in each host crate, the corresponding shrink of `StoreContext` derive
> expansion, and deletion of all `#[macro_export] omnia_wasi_view!` macros.

## 1. Abstract

Every WASI host crate today exports a `macro_rules! omnia_wasi_view` that implements
that crate's per-store view trait on the concrete `StoreCtx`. The
`#[derive(StoreContext)]` proc-macro (`crates/runtime-macro/src/store_context.rs`)
invokes it once per `#[wasi(...)]` field:

```rust
omnia_wasi_blobstore::omnia_wasi_view!(StoreCtx, blobstore);
```

This design retires those macros. Each host crate instead defines a local
`*Parts` trait and a blanket `*View` impl that constructs the view from disjoint
field borrows. The derive emits only a trivial field projection — no view logic,
no hardcoded `core` field name.

## 2. Problem

### 2.1 What `omnia_wasi_view!` does today

Each of the 12 host crates exports an identically named macro. A typical body
(`wasi-blobstore`):

```rust
#[macro_export]
macro_rules! omnia_wasi_view {
    ($store_ctx:ty, $field_name:ident) => {
        impl omnia_wasi_blobstore::WasiBlobstoreView for $store_ctx {
            fn blobstore(&mut self) -> omnia_wasi_blobstore::WasiBlobstoreCtxView<'_> {
                omnia_wasi_blobstore::WasiBlobstoreCtxView {
                    ctx: &mut self.$field_name,
                    table: &mut self.core.table,
                }
            }
        }
    };
}
```

The derive delegates to it:

```rust
let host_views = wasi_fields.iter().map(|(field, path)| {
    quote! {
        #path::omnia_wasi_view!(#struct_ident, #field);
    }
});
```

### 2.2 Why the macro exists

Three real constraints (see [runtime-interface.md §6](runtime-interface.md)):

1. **Fixed accessor signature.** `WasiBlobstore: HasData<Data<'a> = WasiBlobstoreCtxView<'a>>`,
   and `add_to_linker` takes `T::blobstore` — a `for<'a> fn(&'a mut T) -> WasiBlobstoreCtxView<'a>`.
   Whatever produces the view must do it from a single `&mut self`.

2. **Borrow checker.** The body needs *disjoint simultaneous* mutable borrows —
   `&mut self.<backend>` and `&mut self.core.table`. A trait accessor like
   `fn core(&mut self) -> &mut Core` borrows all of `self`, conflicting with the
   backend borrow. Disjoint-field borrows only work through a concrete place
   expression — `self.core.table` alongside `self.<field>`.

3. **Cross-crate knowledge transfer.** The body needs the host crate's trait name
   (`WasiBlobstoreView`), method (`blobstore`), and view type, but it must be
   *expanded in the consumer crate* where the concrete fields live. A
   `macro_rules!` is a compile-time carrier of that per-crate vocabulary across
   the crate boundary.

The macro is not gratuitous — something must carry per-crate names into
consumer-side code. The question is whether a macro is the best carrier.

### 2.3 Why it nonetheless feels like a hack

| Issue | Detail |
|---|---|
| Hardcoded `self.core.*` | The derive parametrizes the core field for its own three impls (`self.#core.wasi`, `self.#core.table`) but does **not** pass that ident to the host macro. Renaming `#[core] base: Core` breaks all 12 host macros silently. |
| Magic name dispatch | Every crate must export a macro named exactly `omnia_wasi_view`. `runtime!` reconstructs paths via `wasi_ident` (`replace("Wasi", "omnia_wasi_")`). The macro name, crate name, and field name are coupled by convention, not by anything the compiler checks. |
| `#[macro_export]` pollution | Twelve macros all named `omnia_wasi_view` are hoisted to crate roots. Path-qualified calls avoid clashes today, but unqualified `use` of two collides. |
| Inconsistency | Most use explicit crate paths (`omnia_wasi_blobstore::…`); `wasi-jsondb` uses `$crate::…`. `$crate` is the more hygienic form. |
| Near-duplication | Nine of twelve bodies are byte-identical modulo three names. |

Not all variance is incidental. Three hosts have genuinely different bodies:

| Crate | Difference |
|---|---|
| `wasi-http` | Builds the view via `self.$field.as_view(&mut self.core.table)` |
| `wasi-model` | Threads `host_dispatch: Arc::clone(&self.core.host_dispatch)` |
| `wasi-config` | Ignores the table entirely; reads only `self.$field_name` |

## 3. Goals

1. **Delete all `omnia_wasi_view!` macros** from host crates.
2. **Move view-construction logic into ordinary, testable trait code** owned by each
   host crate.
3. **Parametrize the core field name** in generated code — no hardcoded `core`
   literal.
4. **No behaviour change** — a pure refactor; `cargo make ci` stays green.
5. **Preserve extensibility** — adding a new host crate remains self-contained;
   no central host registry required.

## 4. Non-goals

- Changing `runtime!` input syntax.
- Changing `RegistryBuilder`, `Compiled`, or guest registry / dispatch machinery.
- Replacing `#[derive(StoreContext)]` itself (it stays; only its host-view
  expansion changes).
- A central `omnia-macros` registry that hardcodes all 12 host shapes (see
  [§8 Alternatives considered](#8-alternatives-considered)).

## 5. Design

### 5.1 The borrow-checker trick

A single method *can* hand back several disjoint borrows at once, because the
borrow checker sees they come from distinct fields of one concrete place
expression. Split `self` exactly once in a method the concrete `StoreCtx`
implements, and never re-borrow `self` afterward:

```rust
pub trait WasiBlobstoreParts: Send {
    type Ctx: WasiBlobstoreCtx + ?Sized;
    fn parts(&mut self) -> (&mut omnia::Core, &mut Self::Ctx);
}
```

`(&mut self.core, &mut self.backend_field)` is allowed — two different fields.
From there, everything else only ever splits `Core` further (also disjoint
fields), so it composes.

A plain accessor approach **does not compile**:

```rust
// DOESN'T COMPILE — `core()` borrows all of `self`
impl<T: HasCore + HasBackend> WasiBlobstoreView for T {
    fn blobstore(&mut self) -> WasiBlobstoreCtxView<'_> {
        let core = self.core();      // &mut *self
        let ctx  = self.backend();   // second &mut *self → conflict
        WasiBlobstoreCtxView { ctx, table: &mut core.table }
    }
}
```

### 5.2 Uniform host shape (e.g. blobstore)

Each host crate defines a local `*Parts` trait and a blanket `*View` impl:

```rust
// in omnia_wasi_blobstore — ordinary, documented, testable code
pub trait WasiBlobstoreParts: Send {
    type Ctx: WasiBlobstoreCtx + ?Sized;
    fn parts(&mut self) -> (&mut omnia::Core, &mut Self::Ctx);
}

impl<T: WasiBlobstoreParts> WasiBlobstoreView for T {
    fn blobstore(&mut self) -> WasiBlobstoreCtxView<'_> {
        let (core, ctx) = self.parts();
        WasiBlobstoreCtxView { ctx, table: &mut core.table }
    }
}
```

The blanket impl is legal because `WasiBlobstoreView` is local to the crate. The
associated type `Ctx: ... + ?Sized` lets the concrete impl name its own field
type; `&mut Self::Ctx` unsizes to the view's `&mut dyn WasiBlobstoreCtx`
automatically (including `Box<dyn WasiBlobstoreCtx>` fields).

### 5.3 What the derive emits

The only generated code is a trivial field projection:

```rust
// generated by #[derive(StoreContext)] for each #[wasi(...)] field
impl omnia_wasi_blobstore::WasiBlobstoreParts for StoreCtx {
    type Ctx = BlobstoreBackend;   // the field's type — parsed from the struct
    fn parts(&mut self) -> (&mut omnia::Core, &mut Self::Ctx) {
        (&mut self.core, &mut self.blobstore)
    }
}
```

Two side effects fall out immediately:

- The `core` field name is parametrized — the derive already binds `#[core]`
  and emits `&mut self.#core`. The hardcoded `self.core.table` literal
  disappears.
- Using `type Ctx = #field_ty` means the derive never has to *name* the ctx
  trait (`WasiBlobstoreCtx`). It uses `Self::Ctx` and the field type it already
  parsed from syn.

The derive must capture the field type in addition to `(field_ident, path)`:

```rust
// store_context.rs — today
let host_views = wasi_fields.iter().map(|(field, path)| {
    quote! { #path::omnia_wasi_view!(#struct_ident, #field); }
});

// store_context.rs — after
let host_views = wasi_fields.iter().map(|(field, path, ty)| {
    quote! {
        impl #path::WasiBlobstoreParts for #struct_ident {
            type Ctx = #ty;
            fn parts(&mut self) -> (&mut ::omnia::Core, &mut Self::Ctx) {
                (&mut self.#core, &mut self.#field)
            }
        }
    };
});
```

The trait name is not uniform across hosts (`WasiBlobstoreParts`,
`WasiModelParts`, …). The derive resolves it by convention: append `Parts` to
the view trait name, or derive it from the `#[wasi(path)]` attribute (see
[§10 Open questions](#10-open-questions)).

### 5.4 Variance cases

The three outliers become ordinary blanket-impl bodies — variance lives in real
code the host crate owns:

**model** (threads `host_dispatch`):

```rust
impl<T: WasiModelParts> WasiModelView for T {
    fn model(&mut self) -> WasiModelCtxView<'_> {
        let (core, ctx) = self.parts();
        WasiModelCtxView {
            ctx,
            table: &mut core.table,
            host_dispatch: Arc::clone(&core.host_dispatch),
        }
    }
}
```

`&mut core.table` and `&core.host_dispatch` are disjoint fields of `*core`.
The existing `impl WasiModelCtx for Box<dyn WasiModelCtx>` forwarding (in
`wasi-model/src/host.rs`) continues to handle boxed backend fields.

**http** (uses `.as_view`):

```rust
impl<T: WasiHttpParts> WasiHttpView for T {
    fn http(&mut self) -> WasiHttpCtxView<'_> {
        let (core, ctx) = self.parts();
        ctx.as_view(&mut core.table)
    }
}
```

**config** (ignores the table):

```rust
impl<T: WasiConfigParts> WasiConfigView for T {
    fn config(&mut self) -> wasmtime_wasi_config::WasiConfig<'_> {
        let (_core, ctx) = self.parts();
        wasmtime_wasi_config::WasiConfig::from(ctx.get_config())
    }
}
```

### 5.5 Naming convention

Each host crate follows a uniform naming scheme:

| Item | Pattern | Example |
|---|---|---|
| Parts trait | `Wasi{Host}Parts` | `WasiBlobstoreParts` |
| Associated type | `Ctx` | `type Ctx = BlobstoreBackend` |
| Method | `parts` | `fn parts(&mut self) -> (&mut Core, &mut Self::Ctx)` |
| View trait | `Wasi{Host}View` | `WasiBlobstoreView` (unchanged) |

The derive references `#path::WasiBlobstoreParts` (or equivalent) via the
`#[wasi(omnia_wasi_blobstore)]` attribute path. No magic macro name required.


### 6.2 `StoreContext` derive expansion (after)

```rust
    #[core]

// generated:
impl omnia_wasi_blobstore::WasiBlobstoreParts for StoreCtx {
```

```rust
struct TestCtx {
    #[core]
```

No manual `WasiModelView` impl required.

## 7. Migration plan

1. **Add `*Parts` trait + blanket `*View` impl** to each of the 12 host crates.
   Land all 12 in one change (mechanical; bodies are small).
2. **Update `store_context.rs`** to emit `Parts` impls instead of
   `omnia_wasi_view!` invocations. Capture field types from syn.
3. **Delete all `omnia_wasi_view!` macros** from host crates.
4. **Update `runtime-interface.md` §4 non-goals** — the "they stay" clause for
   `omnia_wasi_view!` is superseded by this design.
5. **Verify** all examples and `cargo make ci` stay green.

Steps 1–3 must land atomically. A partial migration (derive emitting `Parts`
impls before host crates provide blanket impls, or vice versa) breaks every
deployment.

## 8. Alternatives considered

### 8.1 Parametrize the existing macro (Option 1)

Pass the `#[core]` ident into `omnia_wasi_view!`:

```rust
#path::omnia_wasi_view!(#struct_ident, #field, #core);
```

Cheap and high value for the hardcoded-`core` problem, but keeps the macro
surface, the magic name dispatch, and the near-duplication across nine uniform
crates. A reasonable incremental step if a full migration is deferred.

### 8.2 Shared helper macro in `omnia` (Option 2)

A generic `omnia::wasi_ctx_view!` the uniform crates forward to. Collapses
duplication but does not eliminate the per-crate macro carrier.

### 8.3 Central `HostParts<H>` trait in `omnia`

```rust
pub trait HostParts<H> {
    type Ctx: ?Sized;
    fn parts(&mut self) -> (&mut Core, &mut Self::Ctx);
}
```

Centralizes the trait definition but requires the derive to know the host
*marker type* (`WasiBlobstore`), not just the crate path. Pushes more onto the
`#[wasi(...)]` attribute. Per-crate `*Parts` naming is simpler for the derive.

### 8.4 Rich `#[wasi(...)]` descriptor attribute (Option 4)

```rust
#[wasi(crate = omnia_wasi_model, view = WasiModelView, method = model,
       ctx_view = WasiModelCtxView, host_dispatch)]
model: Box<dyn WasiModelCtx>,
```

Removes exported macros entirely but pays in attribute verbosity, more parsing
in `runtime-macro`, and host crates losing local ownership of view shape.

### 8.5 Central host registry in `omnia-macros`

Hardcode all 12 host shapes in one proc-macro crate. Breaks extensibility —
adding a 13th host requires editing the central crate. Rejected.

## 9. Tradeoffs

### 9.1 Blanket impls foreclose bespoke view impls

Once `impl<T: WasiBlobstoreParts> WasiBlobstoreView for T` exists, no type can
implement `WasiBlobstoreView` except through `Parts` (coherence). Today's macro
impls the view per-type, so hand-written runtimes can hand-roll a view. Under
this design they must go through `Parts` or the derive. For a design where
there is one `StoreCtx` per runtime this is arguably an improvement; it removes
an escape hatch.

### 9.2 Testability is real but uneven

You can unit-test a blanket impl with a stub `Parts` type (`Core` fields are
`pub`). The win concentrates in `wasi-model` (`host_dispatch` threading) and
`wasi-http` (`.as_view`). For the nine uniform crates the body is a two-field
projection; the trait + associated-type + `?Sized` machinery is more ceremony
than the macro it replaces, for little testability gain.

### 9.3 Pragmatic hybrid

Apply the `Parts` pattern only where the body has behaviour (`wasi-model`,
`wasi-http`). Keep a cleaned-up, `core`-parametrized macro for the nine trivial
uniform crates. Less uniform, but minimizes ceremony where the macro body is
already one line of logic.

## 10. Open questions

- **Trait name resolution.** Should the derive append `Parts` to a view trait
  name derived from the crate path (e.g. `omnia_wasi_blobstore` →
  `WasiBlobstoreParts`), or should `#[wasi(...)]` accept an explicit trait path
  (e.g. `#[wasi(omnia_wasi_blobstore::WasiBlobstoreParts)]`)?
- **Hybrid vs uniform.** Apply `Parts` everywhere, or only to `wasi-model` and
  `wasi-http` initially?
- **`wasi-config` long-term.** Its view impl ignores `Core` entirely. If more
  hosts follow that shape, consider a `#[wasi(..., no_core)]` attribute or a
  separate derive path. Not needed for the initial migration — config still
  implements `Parts` with an ignored `_core` binding.

## 11. Acceptance criteria

1. All 12 host crates define a `*Parts` trait and blanket `*View` impl; no
   `omnia_wasi_view!` macros remain.
2. `#[derive(StoreContext)]` emits `Parts` impls with parametrized core field
   name and captured field type.
3. `replay.rs`, `linking.rs`, and all examples build and run unchanged from the
   caller's perspective.
4. `cargo make ci` stays green.
5. No behaviour change: view construction, resource table threading, and
   `host_dispatch` cloning are identical before and after.

## 12. Risks and invariants

- **Atomic landing.** Derive + all 12 host crates must flip together (same
  hazard as [runtime-interface.md §12](runtime-interface.md)).
- **Coherence.** Blanket impls mean all `*View` implementations route through
  `*Parts`. Document this for authors of hand-written runtimes.
- **Law 2 preserved.** `Parts` and view construction carry no consumer
  vocabulary; `host_dispatch` remains the generic host→guest seam.
- **Instance-per-call unchanged.** View construction runs per `store()` call;
  `build_store` / `instantiate` defaults are untouched.
